# Concrete Item Scope

This is the long-run scope ledger for Concrete's foundation, primitive, and component standardization work. Read `CODE.md` first, then this file, then `MIGRATION_QUEUE.md`, then `REFACTOR_RUNBOOK.md`.

The ledger is intentionally operational. It records what each item owns, what it must not own, and what migration state it is in. Update it in the same chunk that changes an item's boundary, styling owner, or dependency shape.

## Layer Contract

Foundations own values and global language. They may define raw values, semantic aliases, reset rules, and tiny utilities. They do not define item selectors.

Primitives own HTML vocabulary. They may render DOM, apply stable Concrete classes, define ARIA and data attributes, and own one scoped `styles.css` file. Primitive CSS must use foundation variables or foundation utilities only.

Components own workflows. They may parse schemas, control state, map data, call utilities, and assemble primitives. The migration target is no component-owned active CSS, no raw styled HTML, and no component selector vocabulary unless explicitly recorded as temporary debt.

## Status Keys

- `foundation-centralized`: Foundation values still live in the root stylesheet.
- `foundation-owned`: Foundation values live in their owning foundation stylesheet.
- `primitive-centralized`: Primitive selectors still live in `styles/primitives.css`.
- `component-centralized`: Component selectors still live in `styles/components.css`.
- `component-assembled`: Component has no active selector debt and assembles primitives/utilities only.
- `component-local-debt`: Active selectors live in the component item stylesheet as explicit temporary debt.
- `internal-primitive`: Private primitive used to retire component-to-component or duplicated JSX.
- `needs-boundary`: The next migration pass must decide whether to create, expand, or promote a primitive before moving styles.
- `owned`: The item owns its selectors in its own `styles.css` and uses foundation values only.

## Chunk Protocol

1. Pull the active chunk from `MIGRATION_QUEUE.md`.
2. Pick one item or one small cluster from that chunk.
3. Read the item `component.tsx`, `schema.ts`, `examples.tsx`, `meta.ts`, and current active CSS.
4. Classify each selector as foundation, primitive, component debt, or dead code.
5. Add missing foundation tokens before moving primitive CSS.
6. Move selectors only into the owning primitive stylesheet, with no raw values.
7. Remove component CSS and raw styled HTML only after the primitive vocabulary exists.
8. Update the item rows below with the new status, boundary decision, and any remaining debt.
9. Run the focused gate, then update `MIGRATION_QUEUE.md` and `REFACTOR_RUNBOOK.md`.

## Foundations

- `foundations/colors` - Owns surface, transparent, ink, foreground, border color, signal, syntax, inverse, data, currentColor wash, chart/diagram/range color semantics, skeleton color semantics, scalar-control track color, and tiny shared color utilities. Status: `foundation-owned`. Boundary: data-tone and diagram-tone utility classes live here because multiple chart/table/diagram renderers share them. Historical component-named color aliases stay here until a later token-tightening pass generalizes or moves them; scalar control tokens use the promoted `stepper-control`/`range-control` primitive names. Example coverage: core palette, signal channels, and generated-output data color order.
- `foundations/elevation` - Owns shadow, ring, filter, stroke-width, border-width depth language, and the shared focus-target utility. Status: `foundation-owned`. Boundary: effects and line-weight tokens live here even when a token name still references a migrated primitive family; `.concrete-focus-target` is foundation-owned because it applies focus-ring language across primitives without becoming button or focus-ring preview DOM. Scalar control shadows use the promoted `stepper-control`/`range-control` primitive names. Example coverage: elevation scale and nested surface stack.
- `foundations/motion` - Owns duration, easing, transform, rotation, scale, progress, opacity, scalar-control motion, and default animated state-position tokens. Status: `foundation-owned`. Boundary: interaction timing, hover opacity, skeleton translation, and motion-state values live here; primitive styles consume them without declaring raw timings. Range-control thumb transform and scale tokens are primitive-named, not component-named. Example coverage: motion token vocabulary and interaction-state timing.
- `foundations/radii` - Owns the radius scale and pill radius. Status: `foundation-owned`. Boundary: all global radius values live here, including small bridge radii needed by existing keycap/button geometry. Example coverage: corner scale and control hierarchy.
- `foundations/spacing` - Owns spatial scale, sizes, measures, templates, offsets, layout grids, z-indexes, responsive layout aliases, visually-hidden utility geometry, control/tooltip/brand sizing, scalar-control geometry, and the responsive root overrides for layout aliases. Status: `foundation-owned`. Boundary: layout values live here; raw media-query thresholds are documented CSS-language exceptions because custom properties cannot parameterize media conditions. Stepper/range control geometry uses promoted primitive vocabulary while compatibility classes may still say number-stepper/range-slider. Example coverage: spacing scale and pressure-density bands.
- `foundations/textures` - Owns grid, lattice, dot, line, dash-array, and repeating/radial texture variables. Status: `foundation-owned`. Boundary: visual pattern values live here while rendered texture DOM remains owned by primitives. Example coverage: texture tokens and diagram coordinate ground.
- `foundations/typography` - Owns font-family variables, display settings, type scale, line heights, weights, tracking, text-decoration aliases, and the shared `.concrete-root` typography utility. Status: `foundation-owned`. Boundary: font imports and `@font-face` declarations remain in `src/styles.css` so public CSS import ordering stays valid; the token values live here. Example coverage: typography roles, scale, and mono trace language.
